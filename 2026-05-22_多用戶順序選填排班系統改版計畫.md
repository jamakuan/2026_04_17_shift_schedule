# 多用戶順序選填排班系統改版計畫

日期：2026-05-22

## 目標
- 將目前「系統自動把日期派給人」的靜態頁流程，改成「系統先算順序與天數，人再依序選填」的多用戶共享系統。
- 第一版採用 `FastAPI + SQLite`，保留現有頁面邏輯作為前端基底，新增最小後端處理登入、共享狀態、流程鎖定、版本紀錄與操作 log。

## 核心規則
- 系統先計算本月順序與每人可選天數。
- 依順位 `1 → 2 → 3 → 4` 選填，每人輪到時一次選完自己的全部日期。
- 下一位一定會看到前一位已選結果，且不能選已被占用日期。
- 全員完成後，是否開放修改由管理者控制。
- 修改期間內，任何登入員工都可改整月所有人的班表。
- 所有修改都要留下完整紀錄。
- 超過截止後只有管理者可修改。
- 登入第一版接受只用員工編號，並明確接受冒用風險，靠 audit log 追查。

## 系統狀態與流程模型
- `draft`：管理者建立月份、系統算出假日清單、順序、每人 quota。
- `picking`：依順位逐位選填，只有當前順位可提交。
- `review_open`：全員完成後，由管理者手動開啟修改期間，所有登入員工可改整月。
- `locked`：截止後鎖定，只有管理者可修改。

### 順序與天數規則
- 假日上班總天數除以 4，餘數依本月順序由前往後各加 1 天。
- 下月順序依本月最終確認結果的實際值班天數排序，少者優先；同天數維持本月相對順序。

## 使用者與權限
### 角色
- `admin`：建立或停用帳號、建立月份、調整順序、代填、修改任意月份、開關修改期間、查看版本與 log。
- `employee`：登入後查看目前月份、在自己輪到時選填、於修改期間內修改整月。

### 登入規則
- 只輸入員工編號即可登入。
- 每次登入、登出、提交選填、修改月份、開關期間、管理者覆寫，都要寫入 audit log。
- log 至少記錄：時間、登入者、動作類型、月份、變更前後摘要、受影響人員。

## 資料模型
- `users`：員工編號、姓名、角色、啟用狀態。
- `schedule_months`：年份、月份、流程狀態、目前輪到誰、修改期間起訖、是否鎖定。
- `month_participants`：該月順序、每人 quota、是否完成初次選填。
- `holiday_dates`：該月所有可值班日期。
- `assignments`：日期對應到哪位員工。
- `schedule_versions`：每次修改產生一個版本快照。
- `audit_logs`：登入與所有變更紀錄。

### 版本規則
- 初次全員選完後產生一個基準版本。
- 修改期間內每次儲存都建立新版本。
- 版本名稱格式預設為 `YYYYMMDD-HHmm_修改者姓名`。
- 管理者可切換查看任一版本，並看出與前一版本差異。
- 第一版的「像 git 一樣視覺化」定義為：版本列表、兩版本比較、顯示哪些日期從誰改到誰、哪些人受影響。

## 前端互動
### 登入後首頁
- 顯示當月狀態。
- 顯示本月順序。
- 顯示每人 quota。
- 顯示已被占用日期。
- 顯示目前輪到誰。
- 顯示自己身分與權限。

### `picking` 階段
- 非當前順位只能看，不能改。
- 當前順位只能選剩餘空白日期，且選取數量必須等於自己的 quota 才能送出。
- 送出後立即鎖定該次結果，系統自動切到下一順位。
- 第二順位之後看到的是最新共享結果，不會看到舊快照。

### `review_open` 階段
- 所有登入員工都可編輯整月所有人的排班。
- 儲存時後端必須驗證：
  - 每個可值班日期只能有一人。
  - 不得分配到非可值班日期。
  - 每位員工最終天數不得與該月規則衝突，除非管理者明確使用覆寫模式。
- 為避免多人同時改寫互蓋，提交時需做版本號或更新時間檢查；送出過期資料時直接拒絕並要求重新載入最新版本。

### `locked` 階段
- 員工唯讀。
- 管理者可進入修改模式，儲存後仍建立新版本與 log。

## API 介面
- `POST /login`
- `POST /logout`
- `GET /me`
- `GET /months/:year/:month`
- `POST /months/:year/:month/start-picking`
- `POST /months/:year/:month/picks/submit`
- `POST /months/:year/:month/review-window/open`
- `POST /months/:year/:month/review-window/close`
- `POST /months/:year/:month/assignments/save`
- `GET /months/:year/:month/versions`
- `GET /months/:year/:month/versions/:versionId`
- `GET /months/:year/:month/diff?from=...&to=...`
- `GET /audit-logs`

### API 行為重點
- `submit` 只接受當前順位本人或管理者代填。
- `save` 在 `review_open` 允許任一登入員工，在 `locked` 只允許管理者。
- 所有寫入 API 都回傳最新版本號，前端下次提交必須帶回去做衝突檢查。

## 驗收條件
- 建立月份後，系統正確算出假日日期、本月順序與每人 quota。
- 第一順位送出後，第二順位讀到最新剩餘日期，且無法選到已占用日期。
- 非當前順位在 `picking` 階段無法提交。
- 全員完成後，月份可由管理者開啟 `review_open`。
- `review_open` 期間任一員工可修改整月，且每次儲存都新增版本與 audit log。
- 員工修改造成重複日期或 stale version 時，後端會拒絕儲存。
- 截止後員工無法再改，管理者仍可改。
- 下月順序以本月最終版本的實際結果重新計算。
- 版本比較可正確顯示日期異動、人員異動與修改者。

## 假設與預設
- 第一版仍以目前 4 位需值班員工為主，但資料模型允許由管理者維護名單。
- `王宏榮` 仍不參與假日值班，週日不排班，週六與平日國定假日需排 1 人。
- 第一版不接公司 SSO、不做密碼、不做 email 驗證。
- 使用只用員工編號登入的冒用風險由需求方接受，系統只提供完整 log 與版本追查。
- 修改期間內允許員工改整月所有人，因此後端必須以版本衝突檢查保證一致性。
- 若某次修改導致人員天數不符原 quota，第一版預設禁止；只有管理者在覆寫模式下可突破此限制，並留下明確 log。
